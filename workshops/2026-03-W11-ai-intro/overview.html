02 — Core Concept
The Three Levels

This is the main idea we want everyone to walk away with. Click each level for what to say.

Level 1
Use Claude for Coding +
Ask questions, generate code, get explanations. Works anywhere, but Claude doesn't know our project, conventions, or architecture.
Ask a question Generate a function Debug an error
"This is where most people start — and it's a perfectly valid way to use AI"
"You can use ChatGPT, Copilot, or Claude in any folder — it'll give you answers"
"But it doesn't know we use Warp tokens instead of hardcoded colors, or that we prefer Swift Testing over XCTest"
Analogy: "It's like hiring a contractor who's great at Swift but has never seen our codebase"
Level 2
Give Claude Context +
CLAUDE.md, AGENTS.md, and rules are read automatically when you run Claude inside the repo. Same question, vastly better answer.
Global ~/.claude/ Repo CLAUDE.md .claude/rules/
"This is where the magic happens — and it takes zero effort from you"
Three layers: Global (your personal prefs), Repo (shared team knowledge in git), Local (your machine-specific settings)
AGENTS.md tells Claude about our architecture, module structure, engineering policies, SwiftLint rules, Warp design system
.claude/rules/ are always loaded — things like our Unleash CLI reference
Key point: "All of this is version-controlled. When we improve it, everyone gets the improvement."
Level 3
Skills, Agents & MCP +
Repeatable workflows defined as Markdown. 17 skills available. Plus MCP connections to Jira, Confluence, Amplitude, Xcode.
/ios-write-unit-test /ios-pr /ios-review-code
"Instead of just giving Claude knowledge, you give it repeatable workflows"
"Skills are Markdown files with step-by-step instructions. We have 17 of them."
Daily: /ios-pre-push, /ios-pr, /ios-review-code, /ios-write-unit-test
Building: /ios-create-new-module, /ios-plan-feature, /ios-warp-design-system
MCP: Jira, Confluence, Amplitude, Xcode — Claude can talk to these tools directly
"You don't need to be at this level today — but knowing it exists matters"
The jump from Level 1 to Level 2 is enormous — and you get it for free just by running Claude inside our repo.
03 — Demo Plan
Two Live Demos

Real files, real skills, real output. Click each demo for the detailed script.

DEMO A Context Changes Everything ~12 min+
Same file reviewed with and without repo context. Show the dramatic difference, then open the hood.
AiApplicationLetterViewModel.swift
Jobs/Sources/JobEasyApply/Internal/Views/AiApplicationLetter/
1 Review from /tmp (generic)
2 Review from repo (context-aware)
3 Open CLAUDE.md + AGENTS.md
4 Show .claude/rules/ and skills/
Step 1 (3 min): Open terminal in /tmp, run claude, paste the file, ask "Review this ViewModel for code quality"
Narrate: "Generic Swift suggestions. No Warp, no module awareness, no project conventions."
Step 2 (4 min): Open Claude Code in the repo: cd ios-app && claude. Ask the same review question pointing to the file.
Narrate: "Now it references AGENTS.md policies, Warp tokens, SwiftLint rules. Same question, completely different answer."
Step 3 (5 min): Open CLAUDE.md → AGENTS.md → show key sections. Then .claude/rules/ and .claude/skills/.
Mental model: CLAUDE.md = entry point, AGENTS.md = policies, rules/ = always-on, skills/ = on-demand
If Claude is slow: switch to backup recording. Key message still lands — "The difference is context."
DEMO B Skills in Action ~13 min+
Run /ios-write-unit-test on a real untested ViewModel. Then demystify the skill file.
UserDetailsFormViewModel.swift
Jobs/Sources/JobProfile/Internal/ViewModel/
1 Show file (89 lines, 0 tests)
2 Run /ios-write-unit-test
3 Show generated tests
4 Open SKILL.md
Step 1 (2 min): Open the file, walk through it — "Validates name, email, phone. Uses Combine. Has zero tests."
Step 2 (5 min): Run /ios-write-unit-test UserDetailsFormViewModel.swift. While it runs, narrate what's happening.
Narrate: "It's reading the skill file, reading AGENTS.md, analyzing the validation logic, generating tests."
Step 3: Show the output. Point out: uses @Test and @Suite (Swift Testing), tests valid and invalid inputs, tests edge cases.
Step 4 (4 min): Open .claude/skills/write-unit-test/SKILL.md. "This is the entire skill. It's a Markdown file."
Key message: "Skills are not magic. They're checklists. Anyone can read, modify, or create them."
Flash the full skill list: daily workflow, building, CI. Mention MCP servers briefly.
If it fails: show backup recording. Open the SKILL.md file anyway — the demystification doesn't need Claude running.
